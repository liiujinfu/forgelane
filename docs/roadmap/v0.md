# v0 Roadmap

This roadmap defines product boundaries for ForgeLane. It is not a task list.
Implementation work should be tracked in GitHub issues and linked back to the
milestone sections here.

## Roadmap Principles

- Ship one observable delivery loop before adding breadth.
- Use GitHub/GitLab as the provider source of truth for work items, branches,
  PRs, reviews, commits, and CI. In ForgeLane docs, GitLab merge requests map
  to the same operator-facing PR concept.
- Store ForgeLane-owned state for runs, control actions, approvals, events,
  logs, workspaces, and artifacts.
- Make PR delivery the unit of completion.
- Keep extension points narrow until the first vertical slice works end to end.

## v0: Single Issue to Draft PR Loop

### Goal

Start from one provider issue, run one coding agent in an isolated workspace,
create or update a draft PR, and expose enough events and controls for a
human to inspect, stop, retry, or request changes. GitHub remains the first
provider path; the current implementation also includes a narrow
host-aware GitLab path for the same delivery loop.

### User Outcome

A user can point ForgeLane at a GitHub issue and watch the agent turn that issue
into a draft PR with commits, logs, status, and a basic control surface.

### In Scope

- GitHub as the first provider, plus minimal GitLab.com and self-hosted GitLab
  support for the same issue-to-draft-PR loop.
- Repository and issue reference import.
- AgentRun lifecycle with explicit states.
- Local or single-node workspace preparation.
- One agent adapter path.
- Branch creation and draft PR creation.
- Event log for run lifecycle, agent actions, commits, PR updates, and control
  actions.
- Session log capture for the agent run.
- Basic run detail view or CLI output showing status, events, logs, branch, and
  PR link.
- Human control actions: stop, retry, request changes, and close.

### Out of Scope

- Independent issue tracker.
- Provider breadth beyond the narrow GitHub/GitLab WorkItem and ChangeProvider
  behavior needed for the first loop.
- Multi-agent planning or worker orchestration.
- Mobile app, VS Code extension, or IM integration.
- Plugin marketplace.
- Multi-tenant organization management.
- Cloud autoscaling runner fleet.
- Advanced policy engine.
- Full CI replacement.

### Acceptance Criteria

- A GitHub or GitLab issue can be referenced as the starting work item.
- ForgeLane can import and cache the provider issue as a WorkItem snapshot.
- ForgeLane can create a planned AgentRun and immutable RunSpec for that issue.
- A workspace is prepared for the target repository.
- The agent can make a commit on a branch.
- A draft PR is created or updated for the branch.
- Run events and agent logs are visible from ForgeLane.
- A human can stop a running AgentRun.
- A human can retry or request changes after a run finishes or fails.
- Provider-owned data and ForgeLane-owned run state are stored separately.

### Current Progress

The current implementation has reached the CLI-first delivery path through
provider-backed draft PR creation:

- `forgelane init` persists an initialized GitHub, GitLab.com, or self-hosted
  GitLab ForgeProject in the instance-global SQLite store at
  `~/.forgelane/forgelane.db`.
- `forgelane work-items import` reads a GitHub, GitLab.com, or self-hosted
  GitLab issue through the provider boundary, stores a cached WorkItem
  snapshot, and appends a compact `work_item.imported` or
  `work_item.refreshed` Event.
- `forgelane work-items show` reads only the cached local snapshot.
- `forgelane runs create` creates a planned AgentRun, succeeded `start`
  ControlAction, immutable RunSpec snapshot, and matching Events.
- `forgelane runs prepare` creates a RunnerJob, leases a Workspace under
  `~/.forgelane/workspaces/run-<id>/`, clones the current repository into
  `repo/`, and records `workspace.allocated` plus `workspace.prepared` or
  `workspace.prepare_failed` Events.
- `forgelane runs execute` invokes the generic command AgentAdapter, captures
  stdout/stderr as log segments, records terminal run outcomes, materializes
  repository changes into commits, creates or claims a ChangeSet, pushes the
  ForgeLane-managed branch through the selected ChangeProvider, and creates or
  updates a draft PR after the first successful pushed commit.
- `forgelane runs start` performs create, prepare, execute, commit
  materialization, branch push, and draft PR delivery in one command.
- No-change runs complete with `repository_delivery.skipped` and do not create
  provider branches, ChangeSets, or draft PRs.
- Missing credentials and provider push or draft PR failures leave
  inspectable ChangeSet state, Events, logs, and retry guidance.
- `forgelane runs show`, `forgelane runs logs`, and
  `forgelane events list --run` inspect local SQLite state and captured log
  artifacts.
- `forgelane runs stop`, `forgelane runs retry`,
  `forgelane runs request-changes`, and `forgelane runs close` record
  auditable ControlActions. Request-changes keeps the ChangeSet active for a
  later retry; close removes it from the active retry target path while keeping
  local evidence inspectable.
- `docs/testing/v0-smoke.md` captures non-mutating GitHub/GitLab smoke paths,
  provider-write smoke steps, and the expected control/evidence checks.

The v0 exit smoke completed on 2026-06-11 against GitHub issue
`github://github.com/liiujinfu/forgelane/issues/38`. ForgeLane created
AgentRun `1`, materialized commit
`github://github.com/liiujinfu/forgelane@bda9d970e0d65351df96462314b8ce3b52a68fe8`,
pushed provider branch `github://github.com/liiujinfu/forgelane/branches/forgelane/issue-38`,
and opened draft PR `github://github.com/liiujinfu/forgelane/pulls/46`.
The provider-verified diff contained only `forgelane-smoke.txt`, and local
evidence reported `draft_open` with `change_set.branch_push_succeeded` and
`change_set.draft_pr_succeeded` Events.

## v0.1: Provider-Backed Review Loop

### Goal

Close the loop between PR review feedback and follow-up agent runs.

### Likely Scope

The v0.1 boundary is tracked in [v0.1.md](v0.1.md). In summary, v0.1 should
make the CLI operator path easier, read provider-backed PR review feedback,
start follow-up AgentRuns against the existing ChangeSet branch, promote run
reports as the main review surface, and surface CI status alongside run status.

## v0.2: Usable Control Surface

### Goal

Make ForgeLane useful for repeated daily use by a small team.

### Likely Scope

- Web run list and run detail pages.
- Basic authentication for a self-hosted instance.
- Repository-level configuration.
- Notification webhooks for run status and approval events.
- Minimal audit trail export.

## Later Directions

- Provider breadth beyond the narrow GitHub/GitLab WorkItem and ChangeProvider
  paths.
- Multiple agent adapters.
- Runner pools and remote execution.
- Mobile/PWA review surface.
- VS Code extension.
- IM integrations.
- Policy plugins.
- Artifact storage backends.
- Organization/team permissions.

## Resolved v0 Decisions

These questions are resolved for the current v0 cut and should be revisited
through ADRs rather than ad hoc roadmap edits:

- First agent adapter: generic command adapter boundary, with Codex CLI as the
  first built-in preset. See [ADR 0002](../adr/0002-generic-command-agent-adapter.md).
- First runner: Go orchestration and local Workspace preparation behind a
  runner boundary, with a future Rust runner daemon still possible. See
  [ADR 0001](../adr/0001-v0-single-node-cli-first.md).
- First control surface: CLI-first over the same control-plane boundaries a
  later API/web surface should use. See [ADR 0001](../adr/0001-v0-single-node-cli-first.md).
- First persistent store: instance-global SQLite current-state tables plus a
  transactional audit log. See [ADR 0007](../adr/0007-v0-uses-transactional-audit-log.md)
  and [ADR 0009](../adr/0009-v0-uses-instance-global-sqlite-state.md).
