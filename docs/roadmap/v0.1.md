# v0.1 Roadmap

This roadmap defines the v0.1 product boundary for ForgeLane. It is not a task
list. Implementation work should be tracked in GitHub issues and linked back to
the milestone sections here.

## Goal

Close the loop between provider-backed PR review feedback and follow-up
AgentRuns.

v0 proved that ForgeLane can turn one provider issue into a branch and draft
PR. v0.1 should prove that ForgeLane can keep that PR moving after human
review or CI feedback, without making ForgeLane the source of truth for issues,
PRs, reviews, or CI.

## User Outcome

An operator can run ForgeLane from a repository checkout, choose an agent-ready
issue, create a draft PR, inspect a clear run report, review provider PR
feedback, start a follow-up AgentRun that incorporates actionable Change
feedback against the same ChangeSet branch, and see the relationship between
feedback, commits, run status, and CI status.

## Problems Solved

- Operators should not need to copy repository URLs or issue numbers from
  GitHub for the common local-repository path.
- Operators should not need to understand WorkItem cache commands as the
  primary workflow surface.
- Human-facing workflows should be oriented around provider Issues and PRs;
  ForgeLane AgentRun and ChangeSet ids remain useful for programmatic access,
  audit trails, and debugging.
- Change feedback should not be copied manually from GitHub into an ad hoc
  prompt.
- Follow-up runs should append commits to the existing PR branch instead of
  starting an unrelated delivery path.
- A run completion summary should use operator-facing language while preserving
  audit evidence.
- Run review should show whether the PR branch is green, red, or still waiting
  on provider CI.

## In Scope

- Bare `forgelane init` can infer the current ForgeProject from the `origin`
  remote when the current directory is inside a supported GitHub or GitLab
  repository checkout.
- A provider-object candidate selection flow lists provider issues that look
  runnable, then imports only the selected WorkItem snapshot when a run is
  created.
- GitHub PR review, requested-changes, and review-comment snapshots can be read
  through the provider boundary for an existing ChangeSet.
- v0.1 Change feedback focuses on actionable follow-up inputs: requested
  changes, non-outdated unresolved inline review comments, and failing CI/check
  summaries for the current PR head.
- Change feedback can trigger a follow-up AgentRun that targets the existing
  active ChangeSet and appends commits to the same provider branch.
- ForgeLane records the relationship between provider feedback, the follow-up
  AgentRun, generated commits, and ChangeSet state.
- `forgelane runs report <run_id>` becomes the operator-facing delivery summary
  for a run, while the audit-oriented evidence concept remains available for
  lower-level inspection.
- GitHub CI/check status for the ChangeSet branch is surfaced alongside run
  status and the run report.

## Out of Scope

- Web run list or run detail UI.
- Provider webhooks.
- Automatic merge.
- Full policy or approval engine.
- Independent issue tracker.
- Broad GitLab review-loop parity before the GitHub path is proven.
- Multi-agent planning or worker orchestration.
- CI replacement or provider-independent CI system.
- Mobile app, VS Code extension, or IM integration.

## Candidate CLI Shape

Exact command names may change during implementation, but the v0.1 operator
surface should be provider-object first:

```bash
forgelane init
forgelane issue list --ready
forgelane issue start 123
forgelane runs report 1
```

After a human requests changes on the provider PR:

```bash
forgelane pr report 46
forgelane pr retry 46
forgelane runs report 2
```

Known issue numbers remain a direct path:

```bash
forgelane runs start 123
```

`pr retry` should read the current actionable Change feedback by default,
record the compact snapshot plus Events, and then create the follow-up
AgentRun. `pr sync-feedback` remains useful for explicit refresh/debugging, but
it should not be required in the happy-path operator flow.

`work-items import` and `work-items show` remain useful for explicit issue
snapshot refresh, cache management, and debugging, but they should not be the
primary operator path. ForgeLane-local AgentRun and ChangeSet ids remain stable
internal handles for scripts, audit, and low-level debugging.

## Command Boundary

- `issue` is the operator-facing entry point for provider issues and starting
  new work.
- `pr` is the operator-facing entry point for provider PR reports, Change
  feedback, and follow-up work on an existing ChangeSet.
- `runs` remains the ForgeLane execution-record surface for run reports,
  logs, retries by run id, and scriptable lifecycle control.
- `work-items` remains an explicit provider issue snapshot/cache surface for
  debugging and maintenance.
- `events` remains the audit log surface.

## Acceptance Criteria

- `forgelane init` succeeds from a supported repository checkout with an
  `origin` remote and reports the detected ForgeProject.
- Bare `init` does not import issues, create AgentRuns, call provider issue
  APIs, or create `forgelane.workflow.json` unless explicitly requested.
- Operators can list candidate provider issues from the issue surface without
  writing WorkItem state.
- Selecting a candidate imports exactly that WorkItem snapshot and creates a
  normal AgentRun.
- ForgeLane can read GitHub review feedback for an existing draft PR ChangeSet
  and persist compact ForgeLane-owned snapshot state plus Events.
- `pr retry <pr_number>` refreshes actionable Change feedback by default before
  creating the follow-up AgentRun, so operators do not need a separate
  `sync-feedback` step in the normal path.
- Synced Change feedback excludes approvals, outdated review comments,
  non-actionable PR discussion, and historical CI failures for superseded heads
  from the default follow-up RunSpec input.
- A follow-up AgentRun can be created from review feedback and targets the
  existing active ChangeSet branch.
- A successful follow-up run appends commits to the existing provider PR branch
  and does not create a second PR for the same active ChangeSet.
- Run report output includes WorkItem, AgentRun status, ChangeSet status,
  provider branch, PR ref, commits, relevant review feedback refs, CI/check
  status, log pointers, and key Events.
- Provider-owned review and CI data remain provider-owned; ForgeLane stores only
  compact snapshots, relationships, run state, ControlActions, Events, logs,
  and artifacts.

## Implementation Slices

1. Improve local setup ergonomics with bare `forgelane init` origin inference.
2. Add provider-object issue candidate listing and selection.
3. Add `runs report` as the operator-facing summary without breaking the
   existing evidence path.
4. Add GitHub review-feedback read models and provider snapshots for
   ChangeSets.
5. Add PR-facing report/retry commands that resolve provider PR numbers to the
   existing ChangeSet.
6. Create follow-up AgentRuns from synced Change feedback against the existing
   ChangeSet branch.
7. Surface GitHub CI/check status in run reports.

## Risks And Guardrails

- Do not let issue discovery become an independent issue tracker. Candidate
  listing is provider read-only until an operator starts a run.
- Do not treat PRs as v0 WorkItems. PRs are ChangeSet delivery artifacts and
  later review/fix inputs.
- Do not let provider review sync mutate provider state.
- Keep retry and request-changes semantics auditable through ControlActions and
  Events.
- Keep `pr sync-feedback` as an explicit refresh/debug command; do not make it
  a required precondition for the normal `pr retry` workflow.
- Prefer compact provider snapshots over full provider payload copies.
- Preserve the v0 CLI path for scripts while improving the v0.1 operator path.
